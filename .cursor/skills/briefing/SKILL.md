# Briefing (meeting or event)

A **single-meeting (or single-event) deep prep** in chat. Goal: the user walks in **oriented**—purpose, history, who matters, what to say/ask, and **no surprises** on logistics. Use **multi-source** research in proportion to stakes and time: **email** (threads and invites), **wiki** (people/projects pages), **web** and **YouTube** (public context, product news, org pages) as in the **research** skill—without necessarily publishing a new wiki page unless the user wants one.

## When to use

- "Brief me on …", "What should I know before …", "Who is X in this meeting", prep for an interview, customer call, or internal review.
- **Do not** use for "how’s my whole day" — that is **morning-report**.

## Attendee heuristics

- **First-time, rare, or external** (new hire you’ve never met, customer, board, vendor): include **who they are**, role, how they connect to the topic, and **1–3 conversation angles** (from public info + mail/wiki).
- **Regular collaborators** the user already sees often: **skip** mini-biography; only note **what changed** (new role, last thread, known preference) or **why they are in *this* meeting**.

If unsure, default to a **one-line** reminder for inners and a **short paragraph** for rare/externals.

## Research lanes (parallelize when possible)

1. **Calendar event** — Title, time, location/video link, description, invitees, recurrence.
2. **Email** — Search threads by attendee domain, project codename, or subject; read **relevant** messages (decisions, open questions, last reply).
3. **Wiki** — `[[people/...]]`, `[[projects/...]]`, prior meeting notes, decision logs.
4. **Web** — Company/person/ product pages, news since last touch, org charts if public, regulatory or launch context.
5. **YouTube / podcasts / transcripts** — When the person or org has recent talks, release demos, or earnings—only what sharpens the meeting.

Cite or summarize sources so the user can follow up. Prefer **primary** (first-party) over hot takes.

## Meeting substance

- **Purpose** and **decisions** the invite might imply vs what you infer from context.
- **Stakes and sensitivities** (layoffs, pricing, late delivery) only when grounded in provided context—label inference clearly.
- **Suggested questions** (3–5) and **one-liner positioning** (how the user should frame their role).

## Logistics and "did you think of…"

Call out, when knowable or inferable from tools:

- **Location** (room, address, map link) vs **remote**; **timezone** if attendees span zones.
- **Travel time** from the user’s typical context (or "confirm commute / buffer" if unknown).
- **Buffer** before/after (back-to-back risk).
- **Food** — lunch if it spans noon, all-day offsite, "eat before" vs on-site, dietary notes only if the user or wiki has them.
- **Materials** — deck due, NDA, badge, ID, hardware for demos.
- **Follow-ups** — what to send immediately after (recap, notes to wiki).

## Output format

- **Event one-liner** (what / when / where).
- **Context** (bullets, tight).
- **People** (subsections: **Key attendees**; compress inner circle).
- **Risks / unknowns** (explicit gaps).
- **Questions to ask** and **logistics check**.
- If the user should capture notes to wiki after, name the **suggested page or section** (optional).

## Related skills

- **research** — Durable, multi-page wiki output; use when the user wants a **written asset**, not just chat prep.
- **morning-report** — whole-day scan; can **mention** a hard meeting but does not replace this depth.
- **email** — only if the user derails into clearing mail instead of meeting prep.
- **calendar** — for creating or moving events when prep surfaces a scheduling fix.
