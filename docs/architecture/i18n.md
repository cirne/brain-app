# i18n architecture (web app)

This document describes how Braintunnel internationalization works in the Svelte web app today.

## Scope

- **In scope:** user-facing web UI copy rendered by `src/client/**` (Svelte components and client-side copy helpers).
- **Out of scope (for now):**
  - server prompt templates (`src/server/prompts/**/*.hbs`)
  - API error payload localization
  - locale switcher UI (the runtime supports language changes, but the app currently defaults to English).

## Stack and core design

We use:

- [`i18next`](https://www.i18next.com/) for translation runtime
- `i18next-browser-languagedetector` for browser language detection
- JSON namespace files under `src/client/lib/i18n/locales/en/`

The integration entry point is:

- [`src/client/lib/i18n/index.ts`](../../src/client/lib/i18n/index.ts)

### Why this shape

This app is Svelte 5, but we do not use a separate Svelte-specific i18n framework. We expose an app-level translate store (`t`) so components can use `$t(...)` directly while keeping `i18next` as the source of truth.

This keeps the model simple:

- one runtime (`i18next`)
- one translation access pattern (`$t('namespace.key')`)
- one file layout (`locales/<lang>/<namespace>.json`)

## Translation namespaces

English source strings are chunked by UI domain:

- `common.json`
- `nav.json`
- `chat.json`
- `onboarding.json`
- `wiki.json`
- `inbox.json`
- `hub.json`
- `search.json`
- `settings.json`
- `cards.json`
- `access.json`

Path:

- [`src/client/lib/i18n/locales/en/`](../../src/client/lib/i18n/locales/en/)

### Key format

Keys in components use dot-paths with namespace prefix:

- `$t('chat.emptyState.prompt')`
- `$t('common.actions.confirm')`
- `$t('access.policyDetailPage.title')`

The i18n helper splits namespace from key and routes to `i18next.t(..., { ns })`.

## Runtime flow

1. `main.ts` calls `initI18n()` before `mount(App, ...)`.
2. `initI18n()` loads all `en` namespace JSON resources.
3. `t` store points to a translate function backed by `i18next.t`.
4. On `languageChanged`/`loaded`, the store is refreshed so components update.

Files:

- [`src/client/main.ts`](../../src/client/main.ts)
- [`src/client/lib/i18n/index.ts`](../../src/client/lib/i18n/index.ts)

## Fallback behavior

`translate(...)` intentionally falls back to:

1. `defaultValue` if provided
2. otherwise the key itself

This prevents `undefined` copy in non-initialized contexts (notably some server-side/unit-test paths that read client copy helpers directly).

## Tests

Client test bootstrap initializes i18n in English:

- [`src/client/test/setup.ts`](../../src/client/test/setup.ts)

This keeps existing Testing Library assertions stable (`getByText(...)` etc.) while migration work happens.

## Using i18n in components

Pattern:

```svelte
<script lang="ts">
  import { t } from '@client/lib/i18n/index.js'
</script>

<button>{$t('common.actions.save')}</button>
```

Guidelines:

- localize visible text, placeholders, `title`, and `aria-*` labels
- prefer a stable key path by domain (e.g. `inbox.messageThread.*`)
- use interpolation for dynamic values (`{{count}}`, `{{status}}`)
- keep behavioral logic unchanged while extracting strings

## Copy quality and style

String extraction is also a copy-quality checkpoint. Use:

- [`docs/COPY_STYLE_GUIDE.md`](../COPY_STYLE_GUIDE.md)

Key policy:

- prioritize user-benefit wording
- avoid internal plumbing/agent jargon
- keep post-onboarding copy deployment-neutral unless behavior actually differs

## Adding a new localized string

1. Choose namespace by feature domain.
2. Add the key/value in `src/client/lib/i18n/locales/en/<namespace>.json`.
3. Replace hardcoded string with `$t('namespace.key')`.
4. If dynamic, pass interpolation options (`$t('key', { count })`).
5. Run:
   - `npm run lint`
   - `npm run typecheck`
   - relevant `vitest` scope (or full `npm test` for broad changes)

## Current limitations

- single shipped locale in repo (`en`)
- no locale-switcher UI yet
- prompt/template localization is intentionally separate from UI i18n
